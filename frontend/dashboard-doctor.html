🔁 Les 3 scénarios Feign + RabbitMQ
Scénario 1 — Création POST /api/ordonnances → Feign → POST /api/notifications + RabbitMQ CREATED

Le médecin crée une ordonnance. L'ordonnance-service la persiste en MySQL, puis appelle synchroniquement le notification-service via Feign Client pour créer la notification (stockée en MongoDB). En parallèle, un event RabbitMQ CREATED est publié de façon asynchrone.

Scénario 2 — Modification PUT /api/ordonnances/{id} → Feign → POST /api/notifications + RabbitMQ UPDATED

Le médecin modifie diagnostic/médicaments. Le patient est notifié via Feign du nouveau diagnostic. RabbitMQ publie l'event UPDATED pour traçabilité asynchrone.

Scénario 3 — Annulation PATCH /api/ordonnances/{id}/cancel → Feign → POST /api/notifications + RabbitMQ CANCELLED

Le médecin annule l'ordonnance. Le statut passe à CANCELLED en MySQL. Feign notifie le patient. RabbitMQ publie CANCELLED. Le notification-service consomme l'event via OrdonnanceEventConsumer.
